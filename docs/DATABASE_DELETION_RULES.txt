              LLM Gateway 最终数据保护准则 - 绝对规则
═══════════════════════════════════════════════════════════════════════════════

【最高准则】

★ 任何人在触及数据库之前，必须确保备份能够成功恢复 ★

这不是建议，不是最佳实践，这是强制性的规则。

═══════════════════════════════════════════════════════════════════════════════

【强制检查清单】

删除数据库中的任何东西前，必须通过以下检查：

□ 1. 备份文件存在
     └─ 命令: ls -lh /opt/llm-gateway/backups/*.sql.gz

□ 2. 备份不超过 24 小时
     └─ 命令: date; ls -lh /opt/llm-gateway/backups/ | head -1

□ 3. 备份文件可以解压
     └─ 命令: gunzip -t /opt/llm-gateway/backups/llm_gateway_backup_*.sql.gz

□ 4. 备份包含有效数据（>500KB）
     └─ 命令: ls -lh /opt/llm-gateway/backups/

□ 5. 备份可以恢复到数据库
     └─ 每周一自动测试恢复

□ 6. 恢复后的数据完整
     └─ 检查用户数、API Keys数等

□ 7. 获得技术负责人明确批准
     └─ 邮件或文档记录

□ 8. 制定完整的回滚计划
     └─ 如何在 15 分钟内恢复

如果上述任何一项不满足，禁止执行任何删除操作。

═══════════════════════════════════════════════════════════════════════════════

【每天必做】

1. 06:00 UTC+8 — 自动生成备份
2. 06:30 UTC+8 — 自动验证备份
3. 每周一 07:00 UTC+8 — 自动测试恢复
4. 每登录一次 — 检查备份状态

═══════════════════════════════════════════════════════════════════════════════

【命令速查】

检查备份状态:
  ls -lh /opt/llm-gateway/backups/ | head -3

验证备份可以恢复:
  /usr/local/bin/require-backup-ok.sh

查看删除审计日志:
  /opt/llm-gateway/check_deletions.sh

查看数据库完整性:
  cd /opt/llm-gateway && docker compose exec -T postgres psql -U gateway -d llm_gateway

═══════════════════════════════════════════════════════════════════════════════

【禁止操作】

这些操作绝对禁止：

  ❌ DELETE FROM users;                    ← 禁止，改用软删除
  ❌ DELETE FROM api_keys;                 ← 禁止，改用撤销
  ❌ TRUNCATE table_name;                  ← 禁止
  ❌ DROP TABLE;                           ← 禁止
  ❌ DROP DATABASE;                        ← 禁止
  ❌ docker compose down -v;               ← 禁止
  ❌ docker volume rm;                     ← 禁止
  ❌ rm -rf /var/lib/docker/volumes/;     ← 禁止

═══════════════════════════════════════════════════════════════════════════════

【硬件级保护】

所有关键数据都受到文件系统级的保护：

  /var/lib/docker/volumes/llm-gateway_pg_data/_data
  └─ 属性: chattr +a (不可删除，仅追加)
  └─ 权限: 700 (仅 root)
  └─ 结果: 即使 root 也无法直接删除

  /opt/llm-gateway/backups/
  └─ 属性: chattr +a (仅追加)
  └─ 权限: 750
  └─ 结果: 只能添加新备份，不能删除

═══════════════════════════════════════════════════════════════════════════════

【如果违反规则会发生什么】

如果有人尝试：

1. 删除未经备份的数据
   → 系统自动阻止
   → 立即告警所有管理员
   → 操作被记录审计日志
   → 严肃处理

2. 删除无法恢复的数据
   → 启动灾难恢复流程
   → 4 小时内恢复服务
   → 完整的事后分析
   → 防止再次发生

3. 忽视备份检查
   → 禁止所有数据库操作权限
   → 进行强制培训
   → 重新获得权限需要管理批准
